## Key operative decisions

1. **Namespace** `Sellinnate\RagEngine`, package `sellinnate/rag-engine`, facade `Rag`.
2. **Envelope encryption** uses AES-256-GCM. A fresh **per-document DEK** is
   generated by the KMS, used to encrypt content, and stored only in **wrapped**
   form. The plaintext DEK is never persisted (data model §8). Crypto-shredding a
   single document = deleting its wrapped DEK (its only copy); shredding a tenant
   = `destroyKey` on the tenant KEK.
3. **LocalKms versioning**: KEK versions are stored oldest-first; new DEKs are
   wrapped with the newest version; unwrap trials newest→oldest so rotation is
   non-destructive (FR-SEC-05). `destroyKey` wipes all versions = strongest data
   erasure.
4. **DriverManager** dispatches on the `driver` field of a named config block
   (not on the block name), so multiple named connections can share a driver.
   `extend()` lets consumers register drivers without forking (FR-EV-04).
5. **Config-cache safety**: no closures in config; a test walks the config tree
   asserting the absence of `Closure` (NFR-CP-02).
6. **Coverage tooling**: the Herd PHP (8.4.22) has no coverage driver. The
   Homebrew PHP (8.4.14) ships Xdebug, so coverage runs as
   `XDEBUG_MODE=coverage /opt/homebrew/bin/php vendor/bin/pest --coverage --min=90`.

## Adversarial review — findings & resolutions

A hostile reviewer audited the security-critical code. Outcome:

**Confirmed correct (no change):** AES-GCM usage (unique IV, pinned tag),
envelope DEK↔ciphertext correspondence, rotation ordering, decrypt-after-shred.

**Fixed this cycle:**

- **H1/M3 — cross-tenant leak via loose `!=`** (`InMemoryVectorStore`). Numeric-string
  tenant ids collided (`'100' == '1e2'`, `'1' == '01'`) and absent keys matched
  `null == ''`. → Switched scalar equality and `eq`/`neq` operators to strict
  `===`/`!==`. Regression tests added.
- **H2 — audit immutability bypassable** via mass-update, `saveQuietly()`,
  `DB::table()`. Model events alone don't cover those. → Added **DB-level WORM
  triggers** (sqlite/mysql/pgsql) in the migration that abort every UPDATE/DELETE.
  Tests cover all bypass paths.
- **H3 — TenantContext singleton bleeds across requests** under Octane/queue. →
  Re-registered as `scoped`; docblock corrected.
- **M2 — rotation test didn't prove which KEK wraps new DEKs.** → Added a
  white-box test that isolates the newest KEK version and asserts the
  post-rotation DEK unwraps under it while the pre-rotation DEK does not.

**Deliberately deferred (tracked):**

- **C1 — mandatory fail-closed tenant scoping.** `InMemoryVectorStore` is a
  primitive (like a raw SQL table): it filters only by what it's given. Mandatory
  tenant scoping sourced from `TenantContext` (fail-closed when absent) belongs to
  the **Retrieval layer (Cycle 3)** and is an explicit obligation there. Documented
  in the `search()` docblock.
- **M1 — crypto-shred tombstone.** A real cloud KMS (AWS/GCP) keeps no app-side
  tombstone after key destruction; preventing silent re-provisioning of a shredded
  tenant id belongs to a **tenant registry / CryptoShredder service (Cycle 4)**,
  not the KMS driver. Current behaviour: `destroyKey` fully erases material (old
  data unrecoverable — the core GDPR guarantee holds).
- **L1–L4** (FileKeyStore chmod race & non-atomic write, corrupt-file masking,
  auto-namespace skips dimension validation, DriverManager stale cache after
  `extend`): low severity, noted for hardening passes.
